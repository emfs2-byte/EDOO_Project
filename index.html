Sobre o Projeto

O SmartHome.cin é um simulador de ecossistema residencial inteligente que demonstra como os pilares da Programação Orientada a Objetos podem ser aplicados para resolver problemas reais de interoperabilidade entre dispositivos IoT heterogêneos. O coração do sistema é um Smart Hub escrito em C++ que gerencia Lâmpadas, Ar-Condicionados e Câmeras de Segurança, expondo uma API REST para que uma interface moderna em React controle toda a residência em tempo real.

📊

PAINEL DE CONTROLE

Visualização em tempo real de todos os dispositivos ativos, com estado sincronizado entre backend C++ e frontend React.

⚡

GESTÃO DE ENERGIA

Cálculo dinâmico do consumo estimado da residência em Watts, atualizado a cada alteração de estado.

❄️

CONTROLE TÉRMICO

Ajuste de temperatura para Ar-Condicionado com validação de intervalo entre 16°C e 30°C no backend.

📷

MONITORAMENTO

Status de gravação REC em tempo real para câmeras, com logs via Polimorfismo no servidor C++.

💡

ILUMINAÇÃO

Controle de on/off para Lâmpadas com feedback visual imediato e logging no terminal do backend.

🔗

API REST

Comunicação via JSON entre React e C++ via rotas GET (status) e POST (toggle, temperatura) com CORS.

Arquitetura // Sistema

Arquitetura de Sistemas Distribuídos em duas camadas: servidor de domínio em C++ e SPA em React comunicando-se via API REST.

⚙️ BACKEND — C++

  • Servidor HTTP com httplib.h
  • Comunicação de baixo nível via Winsock2
  • Estado dos dispositivos em memória RAM
  • Estruturas com std::vector
  • Smart Pointers com unique_ptr
  • Polimorfismo via virtual + override

⚛️ FRONTEND — React

  • Build com Vite
  • Componentes funcionais + useState / useEffect
  • Renderização condicional por tipo de dispositivo
  • Estilização com Tailwind CSS
  • Ícones com Phosphor Icons
  • Dark mode nativo

Modelagem // POO

A robustez do sistema baseia-se na aplicação rigorosa dos quatro pilares da Programação Orientada a Objetos.

01 · ABSTRAÇÃO E HERANÇA

A classe Dispositivo é base abstrata que encapsula comportamentos comuns. As subclasses Lampada, ArCondicionado e Camera especializam as funcionalidades — controle de iluminação, lógica de temperatura (16°C–30°C) e monitoramento com estado de gravação.

dispositivo.h — classe abstrata base
class Dispositivo { private: int id; // acessível apenas via getId() bool ligado; // modificado só por setLigado() protected: std::string nome; // legível pelas classes filhas public: virtual void ligar() = 0; virtual void desligar() = 0; virtual std::string paraJSON() = 0; // ponte com o React virtual ~Dispositivo() = default; };

02 · POLIMORFISMO

Dinâmico: Métodos virtual e override permitem que o servidor percorra a lista genérica de dispositivos invocando a implementação correta em runtime. Downcasting: Na rota de temperatura, dynamic_cast<ArCondicionado*> verifica com segurança se o dispositivo suporta controle térmico.

server.cpp — downcasting seguro
Dispositivo* dev = memoria.obterDispositivoPorId(id); // Tenta o downcast — retorna nullptr se não for ArCondicionado ArCondicionado* ar = dynamic_cast<ArCondicionado*>(dev); if (ar) { ar->setTemperatura(novaTemp); res.set_content("{\"status\":\"sucesso\"}", "application/json"); } else { res.status = 400; }

03 · ENCAPSULAMENTO

Atributos como id, ligado e temperatura são private, acessíveis apenas via Getters públicos e métodos de controle internos. Isso garante que o Ar-Condicionado nunca aceite valores fora de 16°C–30°C, evitando estados inconsistentes.

ar_condicionado.h — regra de negócio encapsulada
void setTemperatura(int t) { if (t >= 16 && t <= 30) { temperatura = t; std::cout << "[AC] Temperatura ajustada para " << t << " Graus\n"; } else { std::cout << "[ERRO] Temperatura " << t << " invalida!\n"; } }

04 · GESTÃO DE MEMÓRIA — C++ MODERNO

Em vez de ponteiros crus, o projeto usa std::unique_ptr e std::move (Smart Pointers). A memória é liberada automaticamente ao remover um dispositivo ou encerrar o servidor, eliminando riscos de Memory Leaks.

gerenciador_memoria.h — vetor heterogêneo com unique_ptr
class GerenciadorMemoria { private: // Vetor que guarda qualquer classe que herde de Dispositivo. // unique_ptr: ao destruir o gerenciador, toda RAM é liberada. std::vector<std::unique_ptr<Dispositivo>> dispositivos; public: void adicionar(std::unique_ptr<Dispositivo> d) { dispositivos.push_back(std::move(d)); } };

2.1 COMPOSIÇÃO

O GerenciadorMemoria detém a posse exclusiva dos objetos via unique_ptr.

2.2 ABSTRAÇÃO

A classe Dispositivo é o contrato base — nenhuma instância direta é criada.

2.3 POLIMORFISMO

O gerenciador trata tudo como Dispositivo*, comportamentos resolvidos em runtime.

Diagrama de Sequência // Fluxo Dinâmico

A "vida" de uma requisição: desde a interação no React até a lógica interna do C++.

✅ CAMINHO DE SUCESSO (16–30°C)

O objeto valida o valor, atualiza o atributo interno e o servidor responde HTTP 200 OK, disparando a re-renderização do React.

❌ CAMINHO DE ERRO (<16 ou >30°C)

O objeto comunica o erro ao servidor que retorna HTTP 400, impedindo estados inconsistentes — o Encapsulamento em ação.

Como Rodar // Setup

1

CLONE O REPOSITÓRIO

git clone https://github.com/emfs2-byte/EDOO_Project.git cd EDOO_Project
2

COMPILE E EXECUTE O BACKEND (C++)

Pré-requisito: GCC / MinGW instalado
# Compilação — linka a biblioteca de sockets do Windows g++ main.cpp -o servidor -lws2_32 # Execute o servidor (porta 8080) ./servidor
3

INICIE O FRONTEND (React)

Pré-requisito: Node.js e npm instalados
# Navegue até a pasta do frontend cd web-ui # Instale as dependências npm install # Inicie o servidor de desenvolvimento npm run dev
4

ACESSE A INTERFACE

# Abra no navegador http://localhost:5173 # API REST disponível em http://localhost:8080/api/dispositivos
